# 影序 Bug 与返工地图

## 一、按阶段定位问题

| 阶段 | 问题 | 表现 | 根因 | 修复或预防 |
|---|---|---|---|---|
| 视觉探索 | 首屏元素太多 | 相机、无人机、稳定器、电脑同时出现，主体不清楚 | 没有先确定唯一视觉叙事 | 首屏只保留相机、内存卡、电脑，用一个贯穿动画讲清流程 |
| 动画实现 | 动画与插图重叠 | 滑动时主体遮挡文字和卡片 | 动画层级、占位区域和滚动区间没有一起设计 | 先划定动画安全区，再做 z-index 和滚动绑定 |
| 网站版本 | 打开过旧版网页 | 用户看到的不是当前设计 | 旧输出文件太多，缺少唯一线上源 | 明确 current 版本，旧版本只归档不作为入口 |
| 导航 | 子页面导航不统一 | 首页、导览、下载页右上角内容不同 | 每页独立复制导航，没有共享组件 | 建立统一导航模板和页面清单 |
| 国际化 | 中文/英文只切换部分页面 | 首页切换后子页面仍是中文，或英文后无法切回 | 语言状态只存在单页脚本 | 使用统一语言状态、统一文案字典和全站回归检查 |
| 图标 | 首页或子页面图标缺失 | 品牌识别不一致 | 页面资源引用不统一 | 统一 icon 文件、路径和页面资源检查 |
| 下载 | 版本和下载链接不同步 | 官网显示旧版本，Release 是新版本 | 发布信息散落在多个 HTML/JS 文件中 | 使用 release manifest，发布前自动检查版本、链接、SHA |
| App 状态 | 新任务继承上一轮进度 | 页面显示旧的 copied/skipped 数字 | 每次运行前没有完整 reset | 每个任务开始先 reset，并加入回归测试 |
| 取消任务 | 未处理文件未被明确统计 | 任务看似结束，但历史里不清楚哪些没备份 | pending 文件没有转为 failed | 取消时显式标记 pending/running 为 failed |
| 历史/报告 | 数据库和报告数字可能不同 | copied/skipped/failed 对不上 | 报告依赖内存统计，历史依赖数据库统计 | 结束时统一从主目标结果和数据库规则生成 |
| 安装包 | App 包结构和便携构建路径不一致 | 第一次启动找不到 Python 或内部依赖 | PyInstaller 输出和 DMG App 结构没有先验证 | 先启动已打包 App，再制作 DMG，再挂载验收 |
| 测试 | 异常测试偏晚 | 很多问题到发布前才暴露 | 先做功能和视觉，后做失败路径 | 从第一版就建立异常测试矩阵 |

## 二、哪些返工是必要的

- 从不同视觉方案中筛选最终方向：属于正常探索。
- 从 SMB 改成影序 YINGXU：属于品牌定位调整，但应在早期冻结。
- 真实 SD 卡验收：属于必要的产品可信度验证。
- P0 异常恢复：属于发布前必须完成的可靠性工作。

## 三、哪些返工本可以减少

1. **多次官网视觉重做**：如果先做低保真线框和首屏叙事，方案 B/C 可以更早收敛。
2. **多次导航和语言修复**：如果一开始做共享导航和统一 i18n，会少很多页面级修补。
3. **多次版本号同步**：如果发布前有一个版本清单，只需更新一次。
4. **过早做移动端/Pocket**：桌面端 P0 还未完全稳定时，移动端会分散注意力。
5. **过早加入 AI、Webhook、云端想法**：这些不是首个发布版本的核心闭环。
6. **依赖截图人工发现问题**：可以先用页面清单、链接扫描和浏览器回归测试发现一部分问题。
7. **边开发边改变品牌和文案**：应先建立产品定位、术语表和文本语气。
