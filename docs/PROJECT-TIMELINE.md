# 影序 YINGXU 项目时间线与返工分析

> 时间依据：用户与 Hermes/Codex 的交接内容、Git 提交记录、网站版本文件、测试记录和发布记录。没有明确证据的地方标为“约”。

## 2026-07-19：接手项目与第一次产品/官网扩展

### 做了什么

- 读取 Hermes 交接文档、网站压缩包和已有 App/DMG 资料。
- 了解摄影师素材整理、设备识别、备份、校验、历史记录等目标。
- 先后出现多个官网视觉方案：Hero 左右布局、全屏 Hero、网格背景、渐变字、功能网格。
- 增加或整理了目录排序、批量事件、限速、Webhook、感知哈希、Lightroom 等功能方向。
- 完成 macOS/Windows 安装说明和基础 App 打包工作。

### 暴露的问题

- 项目一开始功能范围过大，AI、云端、移动端、桌面端同时推进。
- 官网视觉方向尚未冻结，导致同一天内产生多个大版本设计。
- 核心用户流程还没有先固化成一张图。

### 重复/不必要工作判断

- 多个视觉方案本身属于必要探索，但如果先制作低保真线框，至少一部分高保真返工可以减少。
- 过早推进高级功能不是 P0 必需项，应该后移。

## 2026-07-20 至 2026-07-21：视觉方向、品牌和动效调整

### 做了什么

- 参考 Flowty 的视觉和滚动叙事方式。
- 设计从相机拔出内存卡、插入电脑、开始备份的贯穿动画。
- 根据截图不断调整动画大小、位置、层级、背景和页面互动。
- 简化首屏主体，最终保留相机、内存卡和电脑。
- 从 Smart Media Backup 逐步形成“影序 YINGXU”品牌方向。
- 调整首屏文案，删除主标题中不合适的逗号、句号等标点。

### 暴露的问题

- 动画初期像静态图片，和页面互动弱。
- 动画主体太小，无法承担页面叙事。
- 动画与页面插图和文字重叠。
- 品牌、标题和视觉方向在实现过程中仍在变化。

### 重复/不必要工作判断

- 动画大小、位置和层级的多次调整是必要的视觉打磨。
- 如果最开始就规定“动画安全区”和“页面内容安全区”，重叠问题可以少出现几轮。
- 标题文字在后期反复修改，说明文案语气和标点规范应该在设计冻结前确定。

## 2026-07-22：从视觉项目转向完整产品

### 做了什么

- 继续完善官网首页、导览页、下载页、安装页、更新日志和 FAQ。
- 增加 Pocket 移动端/PWA 方向和桌面配对思路。
- 增加评论、共创墙、邮箱反馈和 GitHub Issue 入口。
- 继续修复官网动画、页面层级和移动端布局。
- 形成 Windows、macOS、移动端、GitHub 等多平台规划。

### 暴露的问题

- 移动端和 Pocket 功能过早进入开发范围。
- 平台规划多于真实可验证能力，容易让官网文案超前于产品实现。
- 桌面端的异常恢复、历史一致性和安装验收还没有先完全冻结。

### 重复/不必要工作判断

- Pocket 方向有产品价值，但在 macOS P0 未完全收口前属于范围扩张。
- 部分移动端官网内容更适合标注“规划中”，不应和当前可用功能混在一起。

## 2026-07-23：官网发布、导航、图标和国际化修复

### 做了什么

- 建立并发布 YINGXU 官网。
- 设置 `yingxu-app.github.io`。
- 统一影序图标和品牌名称。
- 修复首页和子页面图标缺失。
- 修复中文/英文切换覆盖不完整的问题。
- 修复子页面右上角导航内容不统一的问题。
- 修复英文切换后无法切回中文的问题。
- 恢复首页右上角中文/英文、下载下拉菜单、介绍导览跳转。
- 让官网从旧仓库地址逐步迁移到新的 GitHub Pages 地址。

### 暴露的问题

- 首页和子页面分别维护导航，产生内容差异。
- 语言状态没有全站统一管理。
- 图标路径和页面资源没有统一检查。
- 官网存在多个历史版本，用户容易打开旧页面。
- GitHub Token 曾出现在对话中，这是严重的安全操作风险。

### 重复/不必要工作判断

- 导航、语言和图标反复修复，是这次最典型的可避免返工。
- 如果最开始使用共享模板、统一 i18n 字典和页面清单，这一阶段可以显著缩短。
- 旧版本保留是必要的；但旧版本应该归档，不应继续作为默认入口。

## 2026-07-24 上午：桌面端可靠性和发布准备

### 做了什么

- 完成相机识别兜底：Leica、Fujifilm、Sony 等目录和文件名提示。
- 修复每次新任务继承旧进度的问题。
- 增加或完善异常测试：无 SD 卡、目标盘满、权限不足、源文件消失、网络通知失败、校验失败、重复备份。
- 完成真实 SD 卡验收：87 个文件、约 8.66 GB，首次复制并校验，第二次跳过已备份文件。
- 准备 v1.0.10 macOS 安装包并更新官网。

### 暴露的问题

- 以前的测试重点偏正常路径，异常恢复验证比较晚。
- 历史记录中 copied 数量曾经没有完整保留。
- 报告数字和历史数据库数字存在不一致风险。
- 取消任务时 pending 文件可能没有被明确标记。

### 重复/不必要工作判断

- 真实 SD 卡验收不是重复工作，是发布可信度所必需。
- 版本 v1.0.5 至 v1.0.10 的连续发布反映了快速迭代，但部分发布可以合并为“测试通过后发布”。
- 相机识别兜底是有价值的增量，不属于无效返工。

## 2026-07-24 下午：P0 可靠性闭环与 v1.0.11

### 做了什么

- 取消任务时将 pending/running 文件标记为 failed。
- 取消、partial、error 状态均保留原始 SD 卡，不允许清理。
- 报告加入 `originals_safe` 和 `can_continue`。
- 历史数据库加入 copied_files 导入和迁移支持。
- 增加取消、异常报告、旧数据库迁移、文件占用、目标盘断开测试。
- 全量 17/17 自动化测试通过。
- 构建 v1.0.11 DMG，SHA-256 为：

  `bf972f37da74ef445ac404aba163d29e16542923a9227874e8a2dac3f466023c`

- 完成 DMG 挂载、App 启动、`/api/status` 版本检查。
- 发布 GitHub Release v1.0.11。
- 发布官网 v33，官网、Release、README 和下载页统一到 v1.0.11。

### 当前仍需现场验证

- 实际拔出物理目标盘。
- 实际锁定文件。
- 系统休眠后恢复。
- Finder 中拖入 Applications 的完整交互流程。

这些项目已经有等价模拟或自动化验证，但不能把模拟说成物理现场验证。

## 总结：这次最值得记住的 8 个教训

1. 先锁定核心用户流程，再设计页面和功能。
2. 先做 P0 闭环，再做移动端、AI 和高级功能。
3. 网站导航、语言和版本信息必须有单一来源。
4. 测试矩阵要和功能开发同时建立。
5. 每个失败状态都要告诉用户“发生了什么、数据是否安全、能否继续”。
6. 每次发布前都要对 App、DMG、官网、Release、README、SHA 做一致性检查。
7. 保留旧版本是正确的，但不能让旧版本继续成为默认入口。
8. 任何 Token、密码和授权码都不能出现在聊天记录、截图或公开仓库中。
