# 影序项目完整流程图

## 一、从想法到发布

```mermaid
flowchart TD
    A[发现真实痛点<br/>摄影素材混乱、备份不可信] --> B[用户画像与场景整理]
    B --> C[产品定位与一句话价值]
    C --> D[需求池分级<br/>P0 / P1 / P2 / P3]
    D --> E[定义核心闭环<br/>插卡→识别→复制→校验→历史→报告]
    E --> F[信息架构与页面地图]
    F --> G[视觉方向与交互原型]
    G --> H[技术方案与数据模型]
    H --> I[实现最小可用版本]
    I --> J[单元测试与接口测试]
    J --> K[主动制造异常]
    K --> L{验收是否通过?}
    L -- 否 --> M[记录 Bug<br/>定位→修复→回归测试]
    M --> J
    L -- 是 --> N[版本冻结与备份]
    N --> O[构建 DMG / 计算 SHA-256]
    O --> P[安装包、App、官网、Release、README 对齐]
    P --> Q[真实用户视角验收]
    Q --> R{线上体验是否通过?}
    R -- 否 --> M
    R -- 是 --> S[发布与收集反馈]
    S --> T[下一版本需求排序]
    T --> D
```

## 二、每个阶段应该留下什么

| 阶段 | 必须产出 | 完成标准 |
|---|---|---|
| 痛点 | 用户场景、真实案例 | 能说清楚谁在什么情况下遇到什么问题 |
| 定位 | 一句话价值、目标用户 | 用户 10 秒内理解产品做什么 |
| 需求 | P0-P3 清单、验收条件 | 每个功能都有“完成”的客观定义 |
| 设计 | 页面地图、组件规范、交互流程 | 视觉、导航、语言状态统一 |
| 技术 | 架构图、数据模型、风险表 | 知道数据如何流动、失败如何恢复 |
| 开发 | 可运行版本、变更记录 | 核心闭环先跑通，不被高级功能拖住 |
| 测试 | 测试矩阵、日志、截图 | 正常路径和异常路径都可复现 |
| 发布 | DMG、SHA、Release、官网 | 所有公开入口指向同一个版本 |
| 复盘 | 时间线、Bug 地图、决策记录 | 下一次能直接复用，而不是重新摸索 |

## 三、影序的核心产品闭环

```mermaid
sequenceDiagram
    participant U as 摄影师
    participant A as 影序 App
    participant C as SD 卡
    participant D as 目标磁盘
    participant H as 历史数据库
    participant R as 备份报告

    U->>A: 插入 SD 卡并创建事件
    A->>C: 扫描文件、读取设备与元数据
    C-->>A: 文件清单、设备、时间、类型
    A->>D: 预检查空间、权限、目标可用性
    A->>D: 复制文件或跳过已验证文件
    A->>D: SHA-256 校验
    A->>H: 写入 copied/skipped/verified/failed
    A->>R: 生成 JSON 报告
    R-->>U: 显示成功、跳过、失败、是否可继续
    U->>A: 只有全部完成且校验通过时清理原卡
```

## 四、发布前的四道闸门

1. **功能闸门**：核心流程可以完成。
2. **安全闸门**：失败时原始数据不会被误删。
3. **一致性闸门**：App、DMG、官网、GitHub、README、SHA 版本一致。
4. **体验闸门**：普通用户能看懂、能操作、能找到结果。

任何一道闸门未通过，都不能称为正式发布。
